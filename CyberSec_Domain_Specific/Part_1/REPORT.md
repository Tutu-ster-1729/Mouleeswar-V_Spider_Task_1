# Foundations of Fuzzing – Part 1

## Objective

The objective of this task was to gain an introduction to coverage-guided fuzzing using AFL++ by analysing and testing a provided archive parser. Before this task, I had little exposure to fuzzing concepts and had never used AFL++ before. The goal was therefore not only to obtain crashes, but also to understand the general workflow involved in fuzz testing.

---

## What is Fuzzing?

Fuzzing is an automated software testing technique in which a program is repeatedly executed using large amounts of unexpected, malformed or randomly mutated inputs in order to discover bugs and vulnerabilities.

Coverage-guided fuzzers such as AFL++ attempt to generate inputs that exercise previously unexplored portions of the program. By observing which inputs reach new execution paths, the fuzzer can gradually explore more of the application's logic.

---

## Understanding the Target

The target program provided in this task was a command-line utility called `xarinfo`, which parses and displays information about a custom archive format.

The source code consisted primarily of:

* `main.c`
* `parser.c`
* `parser.h`
* `utils.c`
* `utils.h`

The parser extracted fields such as:

* Archive ID
* Version
* Flags
* Metadata
* Chunk information

Before fuzzing, I first spent time understanding the expected structure of the XAR format by reading the source code and identifying the fields required for a minimally valid archive.

---

## Building the Target

The target was first compiled normally to understand its behaviour using valid inputs.

Afterwards, it was rebuilt using AFL++ instrumentation so that the fuzzer could observe code coverage during execution.

AddressSanitizer (ASan) builds were also generated to assist with reproducing and analysing crashes discovered during fuzzing.

---

## Seed Corpus Creation

Since AFL++ performs better when given meaningful starting inputs, a small seed corpus was manually created.

### Minimal Archive Seed

A minimal valid archive was generated containing:

* Valid magic bytes
* Supported version information
* Zero chunks
* Empty metadata

This served as the simplest valid input capable of passing the initial parser checks.

### Metadata Seed

A second seed archive was created containing metadata fields such as:

* Creator information
* Comment offsets and lengths

This encouraged AFL++ to explore deeper parsing logic related to metadata handling.

Although the corpus was very small, it allowed AFL++ to mutate structurally valid inputs instead of starting entirely from random data.

---

## Fuzzing Procedure

The fuzzing process involved:

1. Building the parser using AFL++ instrumentation.
2. Creating the input seed directory.
3. Launching AFL++ using the seed corpus.
4. Monitoring the AFL++ dashboard for:

   * Corpus growth,
   * Cycle completion,
   * Crashes,
   * Coverage improvements.

The fuzzer was allowed to run until corpus growth stabilised and no substantial increase in discovered paths was observed.

---

## Results

During fuzzing, AFL++ successfully generated malformed archive files that triggered abnormal behaviour within the parser.

The outputs included:

* Saved crash-inducing inputs,
* Expanded test cases,
* Additional execution paths not originally considered.

Two distinct crashes were observed.

The crashes were reproduced manually outside AFL++ by executing the generated crash files against the parser.

---

## Crash Analysis

### Segmentation Fault

One of the generated inputs caused the parser to terminate with a segmentation fault.

This demonstrated how malformed input could cause invalid memory access during parsing.

### Stack Smashing Detection

Another generated input resulted in stack protection mechanisms detecting memory corruption and terminating execution.

Although the exact root causes were not further modified or patched as part of this task, reproducing the crashes independently helped reinforce the importance of validating AFL++ findings.

## Observation

This task served as my introduction to practical fuzzing.

Some of the concepts I observed include, though i may not completely understand them yet:

* The importance of understanding input formats before fuzzing.
* Why even a small seed corpus can significantly improve effectiveness.
* How coverage-guided fuzzers differ from purely random testing.
* The usefulness of AddressSanitizer in crash analysis.
* The need to manually reproduce crashes generated by automated tools.

Initially, the AFL++ dashboard and terminology were unfamiliar, but gradually interpreting the statistics became easier through experimentation.

---

## Limitations

* The parser itself was not patched after crashes were discovered.
* The seed corpus contained only a few manually created examples.
* Fuzzing duration was limited by the scope of the task.
* Deeper root cause analysis of the discovered crashes was not extensively performed.

---

## Conclusion

This task provided a practical introduction to fuzz testing using AFL++. By analysing the input format, constructing a seed corpus and reproducing crashes generated by the fuzzer, I gained a better understanding of how automated vulnerability discovery tools operate in practice.

Prior to this exercise, I had little experience with fuzzing. Completing this task helped me understand the overall workflow involved in preparing targets, generating test cases, interpreting fuzzer feedback and validating discovered crashes.
